# L.E.O. — Persistent Memory for AI Coding Assistants

**Your AI assistant forgets everything between sessions. L.E.O. fixes that.**

Every new session starts from zero — you re-explain your stack, your conventions, your decisions. The more complex the project, the more time you waste repeating yourself.

L.E.O. (**Living Ecosystem Orchestrator**) is an open-source CLI that gives your AI assistant a structured knowledge base. Decisions, rules, patterns, and specialist profiles — all stored in your project, loaded automatically, and evolving with every session.

```
Monday without L.E.O.:              Monday with L.E.O.:

"We use Go with Cobra for the CLI"     "Add the export command."
"Tests go in the same package"         → AI already knows the stack,
"Don't mock the database"               the conventions, and the
"JWT auth, not sessions"                 decisions from last week.
"The deploy target is AWS Lambda"
```

Built in Go. Works with Claude Code today, extensible to other runtimes. Self-hosting on its own repo since v0.2.0.

## Quick Start

```bash
# 1. Install
git clone https://github.com/vmarinogg/leo-core.git
cd leo-core/cli
make install

# 2. Initialize in your project
cd your-project
leo init

# 3. That's it. Your AI now has persistent memory.
```

> Binary downloads for macOS/Linux available on [GitHub Releases](https://github.com/vmarinogg/leo-core/releases). Homebrew tap coming soon.

## How It Works

L.E.O. creates a `.leo/` directory in your project — a runtime-agnostic knowledge base that your AI loads at every session start.

```
your-project/
├── .leo/                          # L.E.O.'s home
│   ├── config.yaml                # your preferences (language, mode, autonomy)
│   ├── profiles/                  # specialist profiles (engineer, architect, CTO...)
│   ├── kb/                        # knowledge base
│   │   ├── docs/                  # JSON documents (decisions, rules, patterns)
│   │   ├── schema.json            # document schema
│   │   └── index.json             # tag-based neural map
│   └── cache/
│
├── .claude/                       # generated by L.E.O. for Claude Code
│   ├── CLAUDE.md                  # auto-generated from .leo/ config + profile + KB
│   └── settings.json              # hooks for automatic KB maintenance
│
└── your code...
```

You write knowledge → L.E.O. validates, indexes, and generates the context file for your AI runtime. Switch runtimes without losing your knowledge.

## What Makes It Different

**Knowledge compounds.** Month 6 is structurally better than month 1. Your AI doesn't just remember facts — it knows the web of decisions behind your codebase. No competitor starting from zero can match 3 months of accumulated context.

**Token economy.** L.E.O. uses a tiered model: expensive models think (orchestration), cheaper models execute (code), scripts handle deterministic work at zero token cost. Same output quality, fewer tokens.

**Specialist profiles.** Instead of one generic assistant, L.E.O. routes tasks to the right specialist profile — backend engineer, security reviewer, architect — each with tailored focus, tone, and context.

```yaml
# .leo/profiles/backend-engineer.yaml
name: Backend Engineer
description: APIs, databases, performance, security
focus:
  - API design and implementation
  - Database modeling and queries
tone: technical, pragmatic, code-first
context_injection: |
  You are operating as a backend engineer specialist.
  Focus on implementation quality and security.
```

**Runtime-agnostic.** Knowledge lives in `.leo/`, not in `.claude/` or `.cursor/`. L.E.O. generates the right context file for each runtime through adapters. Your knowledge is yours, not locked to a tool.

## Commands

| Command | What it does |
|---------|-------------|
| `leo init` | Interactive onboarding — runtime, language, mode, profile, autonomy |
| `leo update` | Sync core rules and profiles from leo-core |
| `leo status` | KB summary — docs, tags, stale count |
| `leo doctor` | Diagnostic checks on `.leo/` health |
| `leo query` | Query by `--tags`, `--type`, `--scope`, `--lifecycle` |
| `leo read <id>` | Read a KB document |
| `leo write <file>` | Write a document (validates + rebuilds index) |
| `leo delete <id>` | Delete a document |
| `leo reindex` | Rebuild index from docs on disk |
| `leo validate` | Validate docs against schema |
| `leo export` | Export KB to portable directory |
| `leo import` | Import KB (merge or replace mode) |
| `leo profile list\|show\|use\|create` | Manage specialist profiles |
| `leo version` | Print version |

## Current Status

L.E.O. is in active development (v0.4.0). It's real, it works, and it's self-hosting — the tool is used to build itself.

What's working:
- Full CLI with interactive onboarding
- JSON knowledge base with schema validation and tag-based indexing
- 16 specialist profiles (11 technical + 5 C-level)
- Claude Code adapter with auto-generated CLAUDE.md
- Communication modes (verbose, concise, caveman)
- Update system for syncing core KB across projects

What's next:
- Homebrew tap for one-line install
- Summary layer for boot optimization (reduce initial context tokens)
- Runtime adapters for Cursor and Windsurf
- KB curator (dedup, merge, pruning)
