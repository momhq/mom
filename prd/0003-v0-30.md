# PRD 0003 — v0.30(alpha): storage consolidation, graph-fluent schema, capture filtering, and tool surface cleanup

## Problem Statement

Developers running `mom` locally as their personal memory layer hit four compounding problems with the pre-v0.30 architecture:

1. **Memories are scattered.** Per-folder `.mom/` directories mean the same memory can exist in multiple project vaults with no single source of truth, backups must cover N directories, and recall has to walk a fragile filesystem chain to surface anything.
2. **Sensitive content leaks into the vault.** Continuous capture writes every turn — including credentials accidentally pasted into a session — to the same store as everything else, with no gate.
3. **Recall quality is dragged down by noise.** Acknowledgements, tool calls, and inner-monologue turns inflate the vault and outscore real signal during recall.
4. **The MCP tool surface is inconsistent and partly obsolete.** Seven tools with mixed naming conventions, two of which (`list_scopes`, `mom_record_turn`) reflect a model the rest of v0.30 is moving away from.

Additionally, `mom lens` — the local dashboard that makes MOM's work tangible — needs centralized log data to render anything useful. Today's per-folder logs make a useful dashboard view essentially impossible.

## Solution

v0.30 consolidates MOM's storage, schema, capture pipeline, and tool surface around a single coherent model:

- **One vault.** `$HOME/.mom/mom.db` is the single canonical SQLite store. Per-folder `.mom/` directories disappear. The JSON-file vault and the `.mom/raw/` transcript log are removed. Logs (`event_log`, `filter_audit`) live in SQLite tables alongside memories. (ADR 0009)
- **Graph-fluent schema.** Tags and entities become first-class node tables joined through `memory_tags` and `memory_entities` edge tables. Tag drift becomes a row-level rename; "everything about X" becomes a cheap join. (ADR 0010)
- **Substance immutability.** A hard boundary at the write API: substance fields (`id`, `content`, `summary`, `created_at`, `session_id`, all `provenance_*`) are never updated. Operational metadata (`type`, `promotion_state`, `landmark`, `centrality_score`, plus edges) is mutable. (ADR 0011)
- **Tulving typology, default-untyped.** `type` is mutable operational metadata, default `untyped`. Capture never assigns a real type; classification happens later via wrap-up suggestions or explicit user/agent action. (ADR 0012)
- **UUID-only IDs.** Every memory has an opaque UUID. Slug-style and session-derived IDs are gone. (ADR 0013)
- **Capture filtering.** Hard filter for secrets (redact-and-persist with `[REDACTED]`, increment `filter_audit` counter, never store matched substance). Soft filter for noise (silent drop). Explicit user writes via `mom_record` bypass both. (ADR 0014)
- **Open provenance vocabularies (UTM-shaped).** Three orthogonal axes — `provenance_actor`, `provenance_source_type`, `provenance_trigger_event` — stored as open strings. Renames are forward-only; new values appear over time and are valid by appearing.
- **Tool surface cleanup.** Five MCP tools with consistent `mom_<verb>` naming: `mom_status`, `mom_recall`, `mom_get`, `mom_landmarks`, `mom_record`. Two obsolete tools removed; one renamed and reimplemented.
- **Bootstrap via global agent-file marker.** With per-folder `.mom/` directories gone, MOM no longer drops a project-local file to signal its presence. Instead, `mom init` writes a sentinel-delimited block (`<!-- mom:start -->` / `<!-- mom:end -->`) at the top of each detected harness's global agent file: `~/.claude/CLAUDE.md` (Claude Code), `~/.codex/AGENTS.md` (Codex), `~/.codeium/windsurf/memories/global_rules.md` (Windsurf), `~/.pi/agent/AGENTS.md` (Pi). All four harnesses concatenate global with project context (no harness discards the global file), so the block reaches the agent on every session in every project. The MCP server also emits an `instructions` field for protocol compliance — no supported harness injects it into agent context today, but the field is correct per spec and self-activates if support lands. (ADR 0015)
- **`mom upgrade` is the migration path.** When crossing the 0.30.0 boundary, upgrade detects legacy vaults, warns, requires explicit confirmation, and supports `--dry-run`. Memories and `.mom/logs/` (from v0.13.0+) are imported into the central database. Legacy directories are left untouched on disk; users delete them when comfortable.

The result: a single backup target, a clean schema, a defensible tool surface, and a `mom lens` dashboard that can finally render the system's work in one view.

## User Stories

1. As a developer using MOM, I want all my memories to live in one place, so that I can back up my entire vault by copying a single file.
2. As a developer, I want recall to find memories regardless of which directory I started the agent from, so that I'm not surprised by missing context.
3. As a developer running `mom upgrade` from a pre-0.30 install, I want to be warned that a one-way migration is required, so that I can decide whether to proceed.
4. As a developer running `mom upgrade --dry-run`, I want to see exactly which legacy vaults will be imported and how many memories each contains, so that I can verify the plan before committing.
5. As a developer who runs `mom upgrade` and changes my mind, I want my legacy `.mom/` directories to remain untouched on disk, so that I can roll back manually if needed.
6. As a developer re-running `mom upgrade` after a successful migration, I want it to be a no-op, so that I don't accidentally double-import.
7. As a developer with sensitive content (API keys, tokens) accidentally pasted into a session, I want MOM's hard filter to redact those substrings before persisting the memory, so that my vault never contains the secret.
8. As a developer auditing what the privacy filter has caught, I want a counter on `filter_audit` showing how often each detection category fired, so that I can verify the filter is working without re-introducing the leak.
9. As a developer running long sessions, I want acknowledgement turns ("ok", "thanks") and tool noise to be silently dropped, so that recall surfaces real signal instead of chatter.
10. As a developer who wants a soft-filtered turn captured anyway, I want to use the explicit-write path (`mom_record`) which bypasses both filter layers, so that the user's intent always wins over heuristics.
11. As a developer using MOM through the Claude Code, Codex, Cursor, Windsurf, or Pi adapters, I want the same five MCP tools available with consistent naming, so that I can predict the tool surface across harnesses.
12. As an agent, I want to call `mom_status` at session start, so that I receive MOM's operating protocol and orient correctly.
13. As an agent, I want to call `mom_recall` with a query and optional tag/session filters, so that I can surface relevant prior context before answering.
14. As an agent, I want to call `mom_get` with a memory ID, so that I can retrieve and cite a specific memory by its identifier.
15. As an agent, I want to call `mom_landmarks` to list landmark memories sorted by centrality, so that I can ground myself in the most-important persistent context.
16. As an agent, I want to call `mom_record` to intentionally save a memory mid-session, so that important moments are preserved even if the watcher has not yet processed the turn.
17. As an agent, I want every memory I create to default to `type=untyped`, so that I am not silently asserting a classification I have no basis to make.
18. As a developer running `mom lens`, I want to see all memories and operational events from a single centralized SQLite database, so that the dashboard shows the complete picture of MOM's work.
19. As a developer running `mom lens` post-upgrade, I want my pre-v0.30 logs to appear in the dashboard, so that the historical record of my MOM activity is preserved through the migration.
20. As a developer on a multi-machine setup, I want my MOM data in a single SQLite file, so that I can sync it through any existing dotfile sync mechanism without coordinating per-folder vaults.
21. As a contributor to MOM, I want substance-immutability enforced at the write API rather than relying on convention, so that no future code path can accidentally rewrite a captured memory's content or provenance.
22. As a contributor adding a new harness adapter, I want to register a new `provenance_actor` value without a schema migration, so that I can ship the adapter without coupling it to a vault upgrade.
23. As a developer running `mom recall` for "everything tagged `recall`", I want a single SQL join rather than a full-text scan over a denormalized tag array, so that recall stays fast as the vault grows.
24. As a developer who renames a tag (e.g., `mcp` → `MCP`), I want the rename to be a row update on the `tags` table, so that I do not have to bulk-rewrite every memory that used the old tag.
25. As a developer running the wrap-up skill, I want untyped drafts surfaced with deterministic type suggestions, so that I can review and classify them without an LLM call.
26. As a developer, I want the wrap-up skill to surface promotion candidates (draft → curated), landmark candidates, and stale draft candidates in addition to type suggestions, so that wrap-up becomes my single curation surface.
27. As a developer migrating from a v0.13.0+ install, I want my historical `.mom/logs/` entries imported into `event_log` with original timestamps preserved, so that my dashboard history is continuous across the upgrade.
28. As a developer who never wants to migrate logs, I want a `--skip-logs` flag on `mom upgrade`, so that I can opt out of log migration when I do not need historical telemetry.
29. As a maintainer, I want UUIDs to be opaque and never derived from content or session, so that the ID cannot be reverse-engineered into structural information about the memory.
30. As a maintainer, I want the old → new ID mapping written to a one-shot log file during `mom upgrade`, so that any rare debugging case where someone needs the legacy ID can be answered without polluting the schema with a `legacy_id` column.
31. As a developer using `mom export`, I want the export envelope to declare a `schema_version`, so that long-lived archives can be re-imported across MOM versions.
32. As a developer using `mom import`, I want it to validate the export's `schema_version` and reject incompatible files with a clear error, so that I do not corrupt my vault.
33. As an agent author, I want adapter allowlists to enumerate the five canonical tool names, so that my harness exposes only the supported surface.
34. As a developer regenerating my project's `CLAUDE.md` (and the equivalent agent files), I want the generator template to emit the v0.30 tool names, so that future sessions see correct guidance.
35. As a developer, I want recall queries with the AND→OR relaxation behavior preserved within the single vault, so that multi-word queries retain their precision-then-recall behavior even though the scope chain is gone.
36. As a developer reading the design doc, I want the provenance vocabularies documented as guides (not enforced enums), so that I understand MOM is open to new actor/source/trigger values without requiring a migration.
37. As a developer, I want `mom_recall` and `mom_landmarks` to no longer accept a `scope` parameter, so that the API matches the actual single-vault model.
38. As a developer running `mom_record` from MCP, I want my memory persisted with `provenance_actor` reflecting my agent and `provenance_trigger_event = 'record'`, so that recall can later attribute the memory back to the intentional-save path.
39. As a developer running the watcher during a live session, I want each captured turn to be persisted with `provenance_trigger_event = 'watcher'`, so that I can distinguish ambient turn-level captures from session-end flushes.
40. As a developer running cartographer-driven `mom map`, I want captured memories stamped with `provenance_actor = 'cartographer'` and `provenance_source_type = 'cartographer-bootstrap'`, so that I can filter them out of "actually-captured-by-an-agent" recall queries.
41. As a security-conscious user, I want the `filter_audit` table to record what categories of secrets were detected and when, but never the matched substance itself, so that the audit trail does not become a secondary leak vector.
42. As a developer with a corrupted or partial `mom upgrade`, I want the operation to be transactional per vault, so that a crash mid-migration does not leave my central database in an inconsistent state.
43. As a developer, I want the upgrade to print a per-vault summary (path, memory count, log entry count, ID remapping count) on completion, so that I can verify nothing was silently dropped.
44. As a contributor, I want each filter pattern (secret regex, noise heuristic) to be configurable in `config.yaml`, so that organizations can tune the defaults without forking.
45. As a developer running `mom lens`, I want filter audit counters visible in the dashboard, so that I can see at a glance whether the privacy filter is active and what it is catching.
46. As a developer opening a fresh session in any project (whether or not I have ever run `mom init` there), I want the agent to receive a "call `mom_status` first" signal automatically via my global agent file, so that MOM bootstraps without per-project setup.
47. As a developer running `mom init` on a new machine, I want the bootstrap block written to the right global agent file for each harness I have installed (Claude Code, Codex, Windsurf, Pi), so that I do not have to configure each one by hand.
48. As a developer running `mom init` more than once on the same machine, I want the global agent-file block to be updated in place (delimited by sentinels) rather than duplicated, so that my agent file does not accumulate copies.
49. As a developer who wants to remove MOM from my system, I want the bootstrap block in my global agent file to be clearly attributed and easy to delete by hand (or via a future `mom uninstall --global`), so that uninstallation is auditable.
50. As a developer with a heavily customized project-level `CLAUDE.md` / `AGENTS.md` / rules file, I want the global bootstrap block to remain in effect because all supported harnesses concatenate global and project context, so that my project customizations do not silently shadow MOM.
51. As a developer running `mom doctor`, I want detection of known harness fragilities — Codex's `AGENTS.md` not-read bug (#8759), Codex's `AGENTS.override.md` shadowing the plain file, Windsurf's intermittent `global_rules.md` load failures — so that I can diagnose why my MOM bootstrap is not firing.
52. As a developer using Pi, I want MOM's tool surface to continue using the Pi extension model (`.pi/extensions/mom-tools.ts`) rather than MCP, because Pi rejects MCP architecturally, so that MOM works in Pi without a protocol mismatch.

## Implementation Decisions

**Module decomposition (10 deep modules).** The implementation is organized around the following modules, each with a simple, testable interface:

- **Vault** — owns the SQLite connection, transactions, and the schema-migration runner. Exposes `Open`, `Migrate`, `Tx`. All other modules use this as their persistence handle.
- **MemoryStore** — CRUD on the `memories` table. Enforces substance-immutability at the API: substance fields are write-once, operational fields are updatable. Exposes `Insert`, `Get`, `UpdateOperational`.
- **GraphStore** — owns `tags`, `entities`, `memory_tags`, `memory_entities`. Exposes upsert/link/list operations and tag/entity rename and merge.
- **Recall** — query engine combining FTS5, AND→OR query relaxation (ADR 0008 preserved intra-vault), and curated/draft tier escalation. No scope parameter. Returns ranked results with citations.
- **Filters** — pure, stateless functions for hard secrets detection + redaction and soft noise classification. Easily unit-testable in isolation.
- **Capture** — orchestrates the filter pipeline and persists the resulting memory. Stamps provenance. Used by the watcher, by hooks, and by `mom_record`. Returns a decision (Written, Redacted, Dropped).
- **Upgrade** — detects legacy `.mom/` directories, plans the migration (used by `--dry-run`), and executes the migration including UUID minting and old→new mapping log. Idempotent.
- **EventLog** — owns `event_log` and `filter_audit`. Exposes `LogEvent`, `IncrementCounter`, `Query`. Consumed by `mom lens`.
- **MCP** — five-tool handler layer. Also returns a short `instructions` field on `initialize` for MCP protocol compliance (no supported harness consumes it for context injection today, but the field is correct and self-activates if support lands). Thin wrappers over the modules above.
- **Bootstrap** — manages the global agent-file marker block, the canonical bootstrap mechanism for v0.30. Detects installed harnesses, locates each harness's global agent file (`~/.claude/CLAUDE.md`, `~/.codex/AGENTS.md`, `~/.codeium/windsurf/memories/global_rules.md`, `~/.pi/agent/AGENTS.md`), and writes / updates the sentinel-delimited block idempotently at the top of the file (after any leading frontmatter or H1 header). Invoked by `mom init` (and a future `mom uninstall --global`). Coordinates with `mom doctor` to detect known harness fragilities (Codex `AGENTS.md` read bug, `AGENTS.override.md` shadowing, Windsurf load failures). (ADR 0015)

**Schema.** The v0.30 `memories` table has the columns defined in the v0.30 design doc and ADRs 0011/0012/0013. `valid_to` is dropped (temporal validity moves to edges in the future relations layer). `type` is mutable with default `untyped`. Provenance vocabularies are open strings — no `CHECK (... IN (...))` constraints. Tags and entities are first-class node tables with edge tables (`memory_tags`, `memory_entities`).

**Provenance vocabularies (UTM-shaped, three orthogonal axes).** Open strings. Documented current values guide adapters and tools but are not DB-enforced. Renames are forward-only. New values appear by virtue of being written. No write-time pair validation between `source_type` and `trigger_event`.

**MCP tool surface.** Five tools: `mom_status`, `mom_recall`, `mom_get`, `mom_landmarks`, `mom_record`. Renames from pre-0.30: `get_memory` → `mom_get`, `list_landmarks` → `mom_landmarks`, `create_memory_draft` → `mom_record`. Removed: `list_scopes`, `mom_record_turn` (folded into `mom_record`). The `scope` parameter on `mom_recall` and `mom_landmarks` is removed. `mom_record` reimplemented to write directly to SQLite (no `.mom/raw/` writes).

**Capture pipeline.** Hard filter detects secrets via a regex pattern library, redacts the matched substring with `[REDACTED]` in the persisted memory, and increments a `filter_audit` counter. Matched substance is never persisted anywhere. Soft filter classifies noise (fluff, tool calls, inner-monologue, file-content code blocks, exact duplicates) and silently drops. `mom_record` (MCP) bypasses both layers — the explicit-write path is the user override.

**`mom upgrade` migration flow.**
- Triggered when the installed version is older than 0.30.0.
- Detects legacy per-folder `.mom/` vaults by walking from `$HOME` and from common project roots.
- `--dry-run` flag prints the migration plan (vault paths, memory counts, log counts, expected UUID remappings) without writing.
- Without `--dry-run`, requires explicit user confirmation before proceeding.
- Imports memories with fresh UUIDs minted (ADR 0013); writes old → new ID mapping to a one-shot log file beside the central database.
- Imports `.mom/logs/` from v0.13.0+ vaults into the new `event_log` table, preserving original timestamps.
- Drops `.mom/raw/`, `.mom/telemetry/`, `.mom/identity.json`, `.mom/schema.json`, and the standalone `.mom/cache/index.db` by not importing them. Legacy directories on disk are left untouched; the user removes them when comfortable.
- `--skip-logs` flag opts out of log migration.
- Idempotent: re-running on an already-migrated source is a no-op.
- Operations are transactional per vault; a crash mid-migration leaves the central database in a consistent state.

**Wrap-up skill expansion.** The session-wrap-up skill expands beyond promotion suggestions to become the canonical local curation surface: surface untyped drafts with deterministic type suggestions, surface promotion candidates, surface landmark candidates, surface stale draft archival candidates. All suggestions are deterministic and human-confirmed.

**Generator templates.** The `CLAUDE.md`, `AGENTS.md`, and per-harness rule files are template-generated. Templates updated to emit the five canonical tool names and the v0.30 trigger vocabulary.

**Adapter and Pi extension impact.** Each adapter's MCP allowlist is updated to the five-tool set. The Pi extension's MCP wrapping is updated in lockstep (tracked as a separate downstream task in the Pi repo).

**Recall behavior changes.** The progressive scope-chain dimension of ADR 0006 is removed (no scopes to escalate across); the curated/draft quality-tier dimension of ADR 0006 stays. ADR 0007 (FTS5 column weights) and ADR 0008 (AND→OR relaxation) continue to apply unchanged within the single vault.

## Testing Decisions

**What makes a good test for this work.** Tests verify external behavior and contracts, not implementation details. A test of `MemoryStore.UpdateOperational` should fail if any substance field becomes mutable, regardless of how the API is implemented. A test of `Filters.DetectSecrets` should pass on a known-good fixture of credential shapes regardless of the regex internals.

**Modules with explicit test coverage commitments:**

- **Filters** — full unit-test coverage of secrets detection on a fixture of credential shapes (provider keys, JWTs, PEM blocks, env-style assignments) and noise classification on a fixture of conversational fluff, tool calls, inner-monologue blocks, and exact duplicates. False negatives on secrets are the highest-cost error MOM can produce; this is where the test budget lives.
- **MemoryStore** — substance-immutability contract tests: every substance field has a test that attempts a write through every public API path and asserts rejection. Tests are the lock that prevents drift.
- **Recall** — AND→OR relaxation behavior, curated/draft tier escalation, FTS5 ranking with the column weights from ADR 0007. Tests verify the contract that single-token queries are unchanged from pre-0.30 and multi-token queries gain precision.
- **Upgrade** — `--dry-run` accuracy (planned counts equal executed counts), idempotency (re-running is a no-op), and crash-resilience (transactional per-vault import). Migration is one-shot in production; the only place to catch defects is here.
- **Capture** — integration tests covering the filter → memory write pipeline end-to-end. Verifies that a turn matching a hard pattern produces a redacted memory plus a `filter_audit` increment, and that a turn matching a soft pattern produces no memory and no audit row. Lighter than Filters' unit tests; verifies orchestration, not pattern correctness.
- **EventLog** — basic insertion test for `event_log`, counter-increment correctness for `filter_audit`, and a query-by-dimension test that the lens dashboard depends on.

**Modules without explicit test mandates:** Vault, GraphStore, MCP. These are exercised through every other test (Vault), through Capture and Recall tests (GraphStore), and through end-to-end MCP integration smoke tests covering tool registration and dispatch (MCP).

**Bootstrap** — idempotent block insertion is the contract that matters. Tests cover: writing the marker block to a fresh global agent file (each of the four supported paths), updating an existing block on re-run without duplication, leaving non-MOM content above and below the block untouched, prepending after a leading YAML frontmatter or H1 header without breaking it, detecting the right per-harness paths on a machine with multiple harnesses installed, and the `mom doctor` checks for the known harness fragilities (Codex `AGENTS.override.md` presence, Windsurf `global_rules.md` reachability).

**Prior art in the repo.** Test patterns from the existing recall, memory schema, and search package tests are the templates to extend.

## Out of Scope

- **Memory↔memory relations (`relations` table).** Sketched in the design doc as the future home for `supersedes` / `contradicts` / `extends` / `references` edges. The graph-fluent schema (ADR 0010) deliberately leaves a clean seat for this; v0.30 ships without it. Lands in v0.X+.
- **Temporal validity on edges (`valid_from` / `valid_until`).** Directly related to the removal of `valid_to` from the memory row (ADR 0011): "this memory is no longer current" semantics belong on edges, not nodes. Lands when relations land.
- **Group entities.** The `entities` table accommodates a `group` value in its `type` column for future use. v0.30 only writes `user` entities; group attribution is not exercised.
- **Multi-machine vault sync.** A natural follow-on to single-file storage (ADR 0009): users can sync `$HOME/.mom/mom.db` through external mechanisms today, but MOM does not ship a sync command in v0.30.
- **Vector indexing / embedding-based search.** Recall in v0.30 stays FTS5-only with AND→OR relaxation. Embedding-based search is deferred — adopted only if real demand emerges.
- **Lifecycle states beyond `draft` and `curated`.** The `promotion_state` enum stays at two values for v0.30. `archived`, `deprecated`, etc., are parked.
- **Per-memory schema-version field.** Migrations rewrite the vault uniformly via `schema_migrations`; individual memories do not carry their own version. Considered and rejected for v0.30.
- **`legacy_id` column on migrated memories.** Old IDs from pre-0.30 vaults are preserved in the upgrade log file only, not as a column on the memory row (ADR 0013).
- **LLM-based capture filtering.** v0.30 capture filtering (ADR 0014) is fully deterministic — pattern-matched secrets and heuristic noise. Semantic duplicate detection, off-topic detection, and other LLM-driven classification are deferred.
- **A user-facing `mom` skill** exposing slash-command surface (`/mom recall`, `/mom wrap-up`, `/mom landmarks`, etc.). Directly related to the bootstrap work in ADR 0015 — the same `mom init` path would install it, and a minimal `/mom boot` would also be the recovery mechanism for the known harness bootstrap fragilities (Codex `AGENTS.md` not-read bug, Windsurf intermittent load failures). The skill design (umbrella vs. per-verb, subcommand dispatch, per-harness packaging, standalone-install-from-marketplace flow) is large enough to deserve its own iteration. Deferred to a post-v0.30 release; manual recovery (user types "call `mom_status`") is the v0.30 fallback.
- **`mom uninstall --global`.** A clean way to remove the bootstrap marker block from each global agent file. Out of scope for v0.30; manual block deletion (the sentinels make this trivial) is the workaround.

## Further Notes

This PRD is the umbrella for v0.30. Six ADRs (0009 through 0014) capture individual hard-to-reverse decisions; the v0.30 design doc carries the integrated picture (schema diagrams, vocabularies, sample data). After this PRD lands on `main`, work will be broken down into tracer-bullet GitHub issues, each linking back to the PRD and its relevant ADRs.

`mom lens` is downstream of this work and one of the primary motivations for centralized log migration: the dashboard becomes the place where users see MOM's work tangibly, and it requires both centralized memories and centralized event logs to render anything useful.

UTM-style open vocabularies for provenance is a deliberate choice: it lets the project add new harnesses, new triggers, and new capture mechanisms without coupling those additions to schema migrations. Adapters and the Pi extension benefit directly — a new harness ships its own actor name and its own trigger conventions without requiring a coordinated MOM release.

ADRs referenced: 0009 (storage consolidation), 0010 (graph-fluent schema), 0011 (substance immutability), 0012 (Tulving typology, default-untyped), 0013 (UUID-only IDs), 0014 (capture filtering), 0015 (bootstrap via MCP instructions and global agent-file marker). Earlier ADRs preserved: 0001–0005 unchanged; 0006 partially superseded (scope-chain dimension removed; quality-tier dimension retained); 0007 and 0008 continue to apply within the single vault.
